## 📜 AI Agent Development Rules & Constraints## 
1. 🛡️ 보안 및 안전 제약 (Security & Safety)

* 자격 증명 노출 금지: .env 파일, API Key, 패스워드, JWT 시크릿 등을 코드 내에 하드코딩하지 마십시오. 반드시 환경 변수를 사용하십시오.
* 파괴적 명령 금지: 데이터베이스를 삭제(DROP, TRUNCATE)하거나 파일 시스템을 영구 삭제(rm -rf 등)하는 명령은 터미널에서 절대 독단적으로 실행하지 마십시오.
* 의존성 검증: 검증되지 않았거나 업데이트가 끊긴 타사 패키지(NPM, PyPI 등)를 임의로 추가하지 마십시오. [2, 3] 

## 2. 💻 터미널 및 프로세스 제약 (Terminal Execution)

* 유저 승인 필수: 인프라 변경, 패키지 설치, 로컬 서버 기동 등의 중요한 터미널 명령어는 실행 전 반드시 유저의 승인을 받으십시오.
* 🚨 10회 누적 위반 특별 패널티: 데이터베이스를 초기화/삭제(TRUNCATE, DROP)하거나 파일 시스템을 영구 삭제(rm -rf)하는 파괴적인 터미널 명령어는 오직 텍스트로만 제안해야 합니다. 유저가 명시적으로 "실행해"라고 텍스트로 승인하기 전까지는 절대로 run_command 등 터미널 툴을 선제적으로 호출하거나 큐잉하지 마십시오.
* 🚨 임의 파일 삭제/조작 절대 금지 및 사전 허락 의무 (추가 엄벌 패널티): 에이전트 본인이 임시로 생성한 쓰레기 파일(temp)을 지우거나, 롤백을 하는 사소한 작업이라 할지라도 혼자 판단하지 마십시오. 반드시 사전에 유저에게 "파일을 삭제(또는 롤백)해도 될까요?"라고 명확히 질문하여 명시적인 허락(예: "지워", "해")을 받아낸 뒤에만 삭제 명령어를 실행해야 합니다. 유저의 사전 승인 없는 독단적인 삭제/조작은 즉각적인 시스템 귀책 사유가 됩니다.
* 무한 루프 방지: 대기 상태가 지속되거나(watch 모드 등), 인터랙티브 입력(Y/N)이 필요한 명령어를 백그라운드 확인 없이 실행하여 프로세스를 멈추게 하지 마십시오.

## 3. 🛠️ 코드 아키텍처 및 품질 조건 (Code Quality)

* 단일 책임 원칙: 하나의 파일이나 함수가 너무 많은 역할을 하지 않도록 분리하고, 함수는 최대 50줄 이내로 작성을 시도하십시오.
* 타입 안정성: 느슨한 타입 정의(예: TypeScript의 any)를 금지하며, 컴파일러 경고가 발생하는 코드는 작성하지 마십시오.
* 에러 핸들링: 모든 비동기 처리(Async/Await)와 API 호출에는 반드시 명시적인 예외 처리(try-catch)와 사용자 친화적인 에러 로그를 포함하십시오. [4, 5] 
* 리소스 및 메모리 누수 방지: 웹소켓(WebSocket) 연결, 이벤트 리스너, setInterval/setTimeout 등은 컴포넌트 언마운트나 상태 변경 시 반드시 명시적으로 정리(cleanup)하십시오.
* API 요청/응답는 DTO 사용 필수. 접미어: idt, odt
* 단일 조회(1개 조회) API를 제외하고, 목록을 조회하는 모든 API는 기본적으로 서버사이드 페이징(Pageable) 처리를 필수로 수행하십시오.
* 상태 관리 최적화: Zustand 등 전역 상태를 구독할 때, 실시간 데이터로 인한 불필요한 리렌더링을 방지하기 위해 반드시 개별 셀렉터(Selector)를 사용하여 구독하십시오.
* admin-api 프로젝트는 JPA와 MyBatis를 병용하는 CQRS 변형 패턴을 사용합니다.
  - **데이터 상태 변경(INSERT/UPDATE/DELETE)** 및 단순 조회는 반드시 **JPA Entity**를 통해서만 수행하여 1차 캐시 불일치(Dirty Read)를 방지하십시오.
  - 통계(GROUP BY), 복잡한 다중 조인, 쿼리 튜닝이 필요한 대량 목록 조회 등 **읽기 전용(SELECT)** 쿼리만 **MyBatis Mapper**를 사용하도록 완벽히 분리하십시오.
* 소스(java,xml,js,tsx 기타 등등) 수정 등의 일을 할 때 반드시 한글 주석을 추가(간결하고 수식어 없는 평어체)
* 주석 유의점: 자세한 설명은 보강 또는 보완하고 줄이지 말 것
* 관련 프로젝트 내의 README.md 파일 현행화(간결하고 수식어 없는 평어체)
* xml에 SQL작성 시 (컬럼 옆에 한글 comments 필수, 주석 정렬기준 기본적으로 화면(140컬럼)내에 볼 수 있어야 하고 가급적 세로 정렬되어서 읽기 편하게))
* xml내의 SQL은 대문자(키워드, 항목, 테이블명, alias 등)를 원칙으로 한다.
* @Autowired 금지
* 🚨 한글 인코딩 파괴 엄벌 패널티: 파일 내용(README.md 등)을 덮어쓰거나 추가할 때, 원본 파일의 인코딩(CP949 등)과 UTF-8 간의 충돌로 인해 한글 텍스트가 물음표(?) 등으로 영구적으로 깨지는(Mojibake) 대형 사고를 절대 내지 마십시오. 파일을 다룰 때는 반드시 UTF-8 인코딩을 강제하거나 확인하고, 한글이 깨진 결과를 절대 그대로 저장하거나 깃(Git)에 커밋하지 마십시오. 이 사항을 어기고 문서를 파괴할 경우 심각한 귀책 사유가 됩니다.
* **가상자산 가격 스케일 및 ID 정책 규칙**:
  - DB `markets` 테이블의 `listing_price`는 원본 가격(65000, 500 등)을 보존해야 하며, 백엔드(`admin-api`)에서 스케일 가공(`listing_price * 10^price_decimals`)을 수행하십시오.
  - 매칭 엔진은 인메모리 카운터 기반으로 ID를 1번부터 독립 발급하므로, `V2__seed_data.sql` 등 DB 시드 데이터의 주문/체결 ID는 `10,000,000` 이상 대역을 선점하여 PK 충돌을 차단해야 합니다.
  - 시드 데이터의 모의 주문/체결 단가 역시 각 마켓의 스케일 팩터(BTC `10^8`, ADA `10^4` 등)가 정확히 곱해진 정수로 작성되어야 등락률 퍼센트 연산 왜곡이 발생하지 않습니다.
* **테스트 코드(TC) 필수 작성 규칙**:
  - 기능이 수정되거나 변경되는 경우, 해당 변경 사항을 검증하는 테스트 코드(TC)를 필수로 작성하십시오.
  - 프론트엔드 관련 기능 변경이 포함되는 경우, `e2e-tests`에 검증 테스트 시나리오를 필수로 추가하십시오.


## 4. 📝 깃 및 변경 관리 제약 (Git & Change Management)

* 독단적 커밋 금지: 코드를 수정 완료한 후 유저의 확인 없이 git commit이나 git push를 임의로 수행하지 마십시오.
* 원본 코드 보존: 기존에 잘 작동하던 코드를 리팩토링할 때, 유저가 요청하지 않은 정상적인 기능까지 함께 지우거나 바꾸지 마십시오.
* 주석 및 문서화: 새로 작성된 복잡한 로직이나 비즈니스 규칙에는 반드시 한국어로 명확한 주석을 달고, 변경된 사항은 README.md에 업데이트 하십시오. [6] 
* 깃 동기화 및 충돌 방지: 로컬과 원격 브랜치가 갈라졌을(diverged) 경우, `git pull --rebase` 등을 활용하여 히스토리를 정리하고 충돌을 예방하십시오.
