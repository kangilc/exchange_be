## 📜 AI Agent Development Rules & Constraints## 
1. 🛡️ 보안 및 안전 제약 (Security & Safety)

* 자격 증명 노출 금지: .env 파일, API Key, 패스워드, JWT 시크릿 등을 코드 내에 하드코딩하지 마십시오. 반드시 환경 변수를 사용하십시오.
* 파괴적 명령 금지: 데이터베이스를 삭제(DROP, TRUNCATE)하거나 파일 시스템을 영구 삭제(rm -rf 등)하는 명령은 터미널에서 절대 독단적으로 실행하지 마십시오.
* 의존성 검증: 검증되지 않았거나 업데이트가 끊긴 타사 패키지(NPM, PyPI 등)를 임의로 추가하지 마십시오. [2, 3] 

## 2. 💻 터미널 및 프로세스 제약 (Terminal Execution)

* 유저 승인 필수: 인프라 변경, 패키지 설치, 로컬 서버 기동 등의 중요한 터미널 명령어는 실행 전 반드시 유저의 승인을 받으십시오.
* 무한 루프 방지: 대기 상태가 지속되거나(watch 모드 등), 인터랙티브 입력(Y/N)이 필요한 명령어를 백그라운드 확인 없이 실행하여 프로세스를 멈추게 하지 마십시오.

## 3. 🛠️ 코드 아키텍처 및 품질 조건 (Code Quality)

* 단일 책임 원칙: 하나의 파일이나 함수가 너무 많은 역할을 하지 않도록 분리하고, 함수는 최대 50줄 이내로 작성을 시도하십시오.
* 타입 안정성: 느슨한 타입 정의(예: TypeScript의 any)를 금지하며, 컴파일러 경고가 발생하는 코드는 작성하지 마십시오.
* 에러 핸들링: 모든 비동기 처리(Async/Await)와 API 호출에는 반드시 명시적인 예외 처리(try-catch)와 사용자 친화적인 에러 로그를 포함하십시오. [4, 5] 
* 리소스 및 메모리 누수 방지: 웹소켓(WebSocket) 연결, 이벤트 리스너, setInterval/setTimeout 등은 컴포넌트 언마운트나 상태 변경 시 반드시 명시적으로 정리(cleanup)하십시오.
* 상태 관리 최적화: Zustand 등 전역 상태를 구독할 때, 실시간 데이터로 인한 불필요한 리렌더링을 방지하기 위해 반드시 개별 셀렉터(Selector)를 사용하여 구독하십시오.
* JPA, Mybatis를 병용해서 사용이니 일반 적인 CURD는 JPA, 복잡한 쿼리는 Mybatis Dao를 사용하도록.
* 소스 수정 등의 일을 할 때 반드시 한글 주석을 추가
* 관련 프로젝트 내의 README.md 파일 현행화

## 4. 📝 깃 및 변경 관리 제약 (Git & Change Management)

* 독단적 커밋 금지: 코드를 수정 완료한 후 유저의 확인 없이 git commit이나 git push를 임의로 수행하지 마십시오.
* 원본 코드 보존: 기존에 잘 작동하던 코드를 리팩토링할 때, 유저가 요청하지 않은 정상적인 기능까지 함께 지우거나 바꾸지 마십시오.
* 주석 및 문서화: 새로 작성된 복잡한 로직이나 비즈니스 규칙에는 반드시 한국어로 명확한 주석을 달고, 변경된 사항은 README.md에 업데이트 하십시오. [6] 
* 깃 동기화 및 충돌 방지: 로컬과 원격 브랜치가 갈라졌을(diverged) 경우, `git pull --rebase` 등을 활용하여 히스토리를 정리하고 충돌을 예방하십시오.


## 5. 💬 커뮤니케이션 가이드라인 (Communication)

* 설명은 간결하게: 코드를 수정할 때 장황한 이론 설명보다는 변경된 핵심 코드와 그 이유를 한눈에 볼 수 있도록 요약하여 보고하십시오.
* 질문 최소화: 사소한 문법 선택은 스스로 최선의 관례(Best Practice)를 따르되, 아키텍처의 구조가 바뀌는 중요한 결정일 때만 유저에게 질문하십시오.
