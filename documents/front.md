# 🌌 JavaF Exchange (HF-X) 프론트엔드 UI/UX 고도화 기획 분석서 (`front.md`)

본 문서는 국내 암호화폐 거래소의 표준인 **Upbit.com(업비트)**과 글로벌 메이저 거래소인 **Binance(바이낸스)**, **Bybit(바이빗)**의 거래 화면 및 웹 프론트엔드 설계를 정밀 분석하여, 우리 **JavaF Exchange (HF-X)** 플랫폼의 사용자 인터페이스(UI) 및 경험(UX)을 차세대 테크 거래소 수준으로 격상하기 위해 도출한 핵심 화면 구성 및 프론트엔드 기술 개발 요건을 수립합니다.

---

## 🏛️ 1. 메이저 거래소 벤치마크 및 화면 분석 (Benchmark Analysis)

### 1-1. Upbit (업비트) - 극강의 간결함과 직관성
*   **레이아웃 특징:** 한국 유저에게 친숙한 **3단 고정 그리드** 구조.
    *   **좌측:** 차트(상단) 및 주문 패널(하단).
    *   **중앙:** 호가창(Orderbook) 중심의 시각적 체결 영역.
    *   **우측:** 마켓 검색 및 시세 리스트.
    *   **하단:** 투자 내역(보유 자산, 미체결 주문, 체결 대기).
*   **UX 강점:**
    *   **호가창-주문 연동:** 호가창의 가격을 클릭하면 즉시 주문 패널의 입력 필드에 해당 가격이 자동 포커스되어 바인딩됩니다.
    *   **자산 즉시 환산:** 가용 자산 대비 비율 패널(10%, 25%, 50%, 100%)을 제공하여 한 손가락 주문 편의성을 극대화합니다.
    *   **주문 정정/취소 간소화:** 미체결 탭에서 별도 화면 이동 없이 즉각 취소(One-click Cancel)가 가능합니다.

### 1-2. Binance / Bybit - 글로벌 프로 트레이더용 초고성능 인터페이스
*   **레이아웃 특징:** 트레이더 취향에 맞춰 위젯 크기 조절 및 컴포넌트 위치 이동이 가능한 **Flexible Grid System(드래그&드롭 리사이징)** 지원.
*   **핵심 기능:**
    *   **호가 병합(Orderbook Grouping):** 0.1, 1, 10, 100 단위 등으로 호가 뎁스 단위를 병합하여 거시적 매수/매도 벽을 파악하는 기능.
    *   **뎁스 차트(Depth Chart):** 매수 잔량과 매도 잔량의 누적 비율을 실시간 산(Mountain) 그래프 형태로 시각화하여 매물대 압력을 한눈에 감지.
    *   **고급 주문 탭:** 지정가/시장가 외에 **Stop-Limit(예약지정가)**, **Market Stop(예약시장가)**, **Trailing Stop(추적손절매)** 등 전문 금융 사양 제공.

## 🛠️ 2. HF-X 프론트엔드 핵심 화면 구성 설계 (Component Architecture)

HF-X에 구축할 최첨단 다크 모드 글래스모피즘 거래소 프론트엔드는 다음 **6대 핵심 컴포넌트**로 구성됩니다.

```
+-----------------------------------------------------------------------------------+
|  [Header] 로고 | 마켓명 (BTC/KRW) | 현재가 | 전일대비 | API Host 설정 | 연결 상태 (Connected) |
+----------------------------------------+------------------------------------------+
|                                        |  [Market List]                           |
|  [Chart Component]                     |  - KRW / USD / BTC 마켓 탭               |
|  - TradingView Lightweight Candlestick |  - 실시간 가격 변동 및 거래대금 정렬       |
|  - 시간 해상도 선택 (1m, 5m, 1h, 1d)    |                                          |
|                                        +------------------------------------------+
|                                        |  [Order Panel]                           |
|                                        |  - 지정가 | 시장가 | 예약주문 탭         |
|                                        |  - 가능 잔액 표시 & 주문 금액 자동 바인딩 |
|                                        |  - BUY (매수) / SELL (매도) 네온 버튼    |
+----------------------------------------+------------------------------------------+
|  [Orderbook & Depth]                   |  [My Orders & Settle Journal]            |
|  - 호가 스크롤 센터 락 & 잔량 비율 바   |  - 보유 자산 실시간 평가액                |
|  - 실시간 체결 강도 (Volume Power)      |  - 미체결 주문 취소 / 누적 체결 이력       |
+----------------------------------------+------------------------------------------+
```

### 2-1. [Component 1] 프로페셔널 차트 엔진 (Candlestick Chart)
*   **기술 스택:** `TradingView Lightweight Charts` (초경량 모던 웹 그래픽스 라이브러리).
*   **화면 요구 사양:**
    *   초당 수십 회 발생하는 실시간 체결 데이터(`trade-events`)를 구독하여 차트의 마지막 캔들 시가/고가/저가/종가(OHLCV) 및 거래량 바를 실시간 버퍼링 업데이트.
    *   보조지표 믹스인: 이동평균선(MA 5, 20, 60, 120), 볼린저 밴드, 그리고 차트 하단의 Volume 지표를 디스크 부하 없이 부드럽게 렌더링.
    *   도구 패널: 타임프레임 전환 스위처(1분, 5분, 15분, 1시간, 1일, 1주일).

### 2-2. [Component 2] 모아보기 및 고정 기능 호가창 (Orderbook)
*   **화면 요구 사양:**
    *   **호가 잔량 비주얼라이저:** 각 호가 라인마다 매수/매도 누적 잔량의 최대치 대비 비율을 반투명 수평 가로 막대 그래프(Visual Depth Bar)로 후방 배치 렌더링.
    *   **센터 로킹 (Scroll Lock):** 현재 최우선 매도호가(Ask)와 최우선 매수호가(Bid)가 만나는 스프레드 영역을 호가창 정중앙에 수평 정렬 고정하고, 마우스 오버 시에만 스크롤이 되도록 설계.
    *   **호가 클릭 액션:** 임의의 호가 라인을 클릭하면 우측 주문 패널의 `주문 가격(Price)` 필드에 해당 가격 값이 0ms 지연으로 자동 복사.

### 2-3. [Component 3] 직관적인 스마트 주문 패널 (Order Panel)
*   **화면 요구 사양:**
    *   **자산 비율 퀵 버튼:** `10%`, `25%`, `50%`, `100%` 버튼 제공. 사용 가능한 KRW/USD 잔액에 맞춰 소수점 8자리까지 정확하게 계산하여 주문 가능 수량(`Quantity`) 자동 연산.
    *   **네온 발광 가시성:** 매수 탭은 하이퍼-비비드 에메랄드 그린, 매도 탭은 강렬한 네온 마젠타 핑크 컬러로 포커싱을 주어 주문 실수 방지.
    *   **주문 전송 피드백:** 주문 버튼 클릭 시 소켓 커맨드 채널(`ws-gateway`)로 즉시 바이너리/JSON 직렬화 패킷을 송출하고, 상단에 세련된 플로팅 토스트 메시지로 체결/접수 현황 브리핑.

### 2-4. [Component 4] 실시간 체결 내역 및 체결 강도 (Trades & Volume Power)
*   **화면 요구 사양:**
    *   **체결 흐름 리스트:** 최근 발생한 체결 가격, 수량, 체결 시간, 매수/매도 구분을 실시간 스크롤 리스트로 출력.
    *   **체결 강도 계측기:** 최근 24시간 또는 최근 1,000건의 `매수 체결량 / 매도 체결량 * 100`을 실시간으로 연산하여 **체결 강도(Volume Power %)**를 하프 도넛 게이지 차트로 출력. 트레이더의 즉각적인 시장 심리 파악 유도.

### 2-5. [Component 5] 보유 자산 & 투자 내역 원장 (Account Ledger)
*   **화면 요구 사양:**
    *   **실시간 평가 손익(PNL):** 백엔드에서 전송되는 현재 마켓 평단가와 내가 매수한 코인들의 수량을 비교하여 실시간 평가금액, 평가손익, 수익률(%)을 초 단위로 재계산 렌더링.
    *   **원클릭 미체결 취소:** 현재 미체결되어 대기 중인 주문 목록을 표시하고, 우측의 `취소` 버튼을 누르면 즉시 백엔드로 CANCEL 요청을 날림.

## 🚀 3. 프론트엔드 성능 최적화 기술 요건 (Frontend Performance Engineering)

초고성능 FIFO 인메모리 엔진의 처리 속도(RTT < 1ms)를 브라우저 화면이 병목 없이 표출하기 위해 반드시 적용해야 할 3대 프론트엔드 기술 스펙입니다.

### 3-1. `requestAnimationFrame` (rAF) 기반 렌더링 버퍼링 (Throttling)
*   **문제점:** 초당 수천 건의 시세 델타 데이터가 소켓으로 유입될 때 브라우저 DOM을 즉각 렌더링(Reflow/Repaint)하면 CPU 팬이 고속 회전하며 화면이 멈추거나 브라우저 탭이 고사하는 병목이 발생함.
*   **해결책:** 웹소켓 수신 모듈에 메모리 큐(Buffer Queue)를 두고, 데이터가 들어올 때 메모리 큐에만 우선 적재한 뒤, 브라우저 주사율(60Hz)에 맞춘 **`requestAnimationFrame` API 주기(약 16.6ms)**마다 누적 델타값을 단 한 번의 DOM 쓰기 연산으로 취합 반영(Batch Render)함.

### 3-2. Virtual Scroll (가상 스크롤) 아키텍처 적용
*   실시간 체결 내역 리스트나 대규모 지갑 입출금 내역 렌더링 시, DOM 엘리먼트 개수가 수만 개로 늘어나면 브라우저 메모리가 고갈됨.
*   현재 사용자의 뷰포트(Viewport)에 노출되는 상하 20~30개의 물리 로우(Row)만 가상 DOM에 유지하고, 스크롤 이동 시 데이터 내용만 갈아끼우는 **가상 스크롤링 기법**을 전면 적용하여 브라우저 가볍기 유지.

### 3-3. WebSocket 연결 이중화 및 오토-커넥트 (Auto-Reconnect)
*   사용자 로컬 브라우저와 Netty 게이트웨이 간 세션 연결이 일시적으로 흔들릴 때, 브라우저가 자동으로 무중단 백오프 재연결을 시도하도록 자바스크립트 소켓 모듈 내에 오토-리커넥터 및 버퍼링 큐 내장.

## 📊 4. 향후 화면 고도화 개발 이행 단계 (Roadmap)

1.  **Phase 1 (Core Integration):** `TradingView Lightweight Charts` 통합 및 웹소켓 델타 시세 바인딩.
2.  **Phase 2 (UX Polishing):** 호가창 센터 락 기능 구현 및 클릭 시 가격 주문창 복사 연동.
3.  **Phase 3 (Optimization):** `requestAnimationFrame` 렌더링 스로틀러 모듈 장착으로 웹소켓 폭주 시의 브라우저 랙(Lag) 100% 소멸.
